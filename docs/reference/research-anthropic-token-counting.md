# Anthropic API Token Counting — Research Reference

## Overview

This document covers all available methods for counting or estimating tokens when working with
the Anthropic API (`@anthropic-ai/sdk`) in TypeScript/Node.js. The primary use-case driving
this research is pre-call token estimation for chunking large source documents in an LLM Wiki
CLI tool — deciding when to split content before sending it to the model.

Token counting matters for three practical reasons:

1. **Avoiding context window overflow** — Claude models have hard input limits. Exceeding them
   causes the API to reject the request.
2. **Cost estimation** — Input and output tokens are billed separately; knowing counts upfront
   lets you budget accurately.
3. **Rate limit management** — Tokens-per-minute (TPM) limits apply alongside requests-per-minute.
   Counting in advance helps pace requests.

---

## Key Concepts

| Term | Meaning |
|------|---------|
| Token | The atomic unit the model processes. Roughly 4 English characters or 0.75 words. |
| Context window | Maximum combined input + output tokens a model accepts in one call. |
| `input_tokens` | Tokens in your prompt (system + messages) actually processed by the model. |
| `output_tokens` | Tokens generated by the model in its response. |
| `cache_creation_input_tokens` | Tokens written into the prompt cache (billed at write rate). |
| `cache_read_input_tokens` | Tokens read from the prompt cache (billed at read rate, much cheaper). |
| BPE | Byte Pair Encoding — the tokenization algorithm Claude (and most modern LLMs) use. |

## Method 1: The Official Token Counting API (Recommended)

### Status

The `POST /v1/messages/count_tokens` endpoint is **generally available** as of 2025. It
**no longer requires** the `anthropic-beta: token-counting-2024-11-01` header that was
mandatory during its beta phase. The SDK method is `client.messages.countTokens()`.

### How It Works

You pass the same payload you intend to send to `client.messages.create()` — including the
model, system prompt, messages array, and tool definitions — and the API returns the token
count without executing the request or consuming compute credits.

**Token counting calls are free.** They are subject to their own separate requests-per-minute
rate limit (independent from the message creation rate limit).

### Basic Usage

```typescript
import Anthropic from '@anthropic-ai/sdk';

const client = new Anthropic(); // reads ANTHROPIC_API_KEY from env

async function countPromptTokens(
  model: string,
  systemPrompt: string,
  userMessage: string
): Promise<number> {
  const result = await client.messages.countTokens({
    model,
    system: systemPrompt,
    messages: [{ role: 'user', content: userMessage }],
  });
  return result.input_tokens;
}

// Example
const tokens = await countPromptTokens(
  'claude-opus-4-5',
  'You are a technical documentation assistant.',
  'Explain the difference between TCP and UDP.'
);
console.log(`Prompt will use ${tokens} input tokens`);
```

### With Tools Included

Tool definitions consume tokens. When your prompt includes tools, pass them to `countTokens`
to get an accurate count that reflects what the actual call will cost.

```typescript
import Anthropic from '@anthropic-ai/sdk';

const client = new Anthropic();

async function countWithTools(
  model: string,
  messages: Anthropic.MessageParam[],
  tools: Anthropic.Tool[]
): Promise<number> {
  const result = await client.messages.countTokens({
    model,
    system: 'You are a helpful assistant.',
    messages,
    tools,
  });
  return result.input_tokens;
}

const tools: Anthropic.Tool[] = [
  {
    name: 'search_docs',
    description: 'Search the documentation index',
    input_schema: {
      type: 'object' as const,
      properties: {
        query: { type: 'string', description: 'Search query' },
        max_results: { type: 'number', description: 'Maximum results to return' },
      },
      required: ['query'],
    },
  },
];

const messages: Anthropic.MessageParam[] = [
  { role: 'user', content: 'Search for information about token limits' },
];

const count = await countWithTools('claude-opus-4-5', messages, tools);
console.log(`Total input tokens including tool definitions: ${count}`);
```

### Pre-Call Chunking Decision Pattern

The canonical use-case for this project: check if content fits before sending.

```typescript
import Anthropic from '@anthropic-ai/sdk';

const client = new Anthropic();

// Model context windows (input limits, conservative — leave room for output)
const MODEL_INPUT_LIMITS: Record<string, number> = {
  'claude-opus-4-5':          180_000,
  'claude-sonnet-4-5':        180_000,
  'claude-haiku-3-5-20241022': 180_000,
};

const RESERVED_OUTPUT_TOKENS = 4096; // budget for the model's response

interface ChunkDecision {
  fitsInContext: boolean;
  inputTokens: number;
  remainingBudget: number;
}

async function checkFitsInContext(
  model: string,
  systemPrompt: string,
  documentChunk: string,
  userInstruction: string
): Promise<ChunkDecision> {
  const messages: Anthropic.MessageParam[] = [
    { role: 'user', content: `${userInstruction}\n\n---\n\n${documentChunk}` },
  ];

  const result = await client.messages.countTokens({
    model,
    system: systemPrompt,
    messages,
  });

  const limit = MODEL_INPUT_LIMITS[model] ?? 100_000;
  const effectiveLimit = limit - RESERVED_OUTPUT_TOKENS;

  return {
    fitsInContext: result.input_tokens <= effectiveLimit,
    inputTokens: result.input_tokens,
    remainingBudget: effectiveLimit - result.input_tokens,
  };
}

// Usage
const decision = await checkFitsInContext(
  'claude-opus-4-5',
  'You are a documentation assistant. Summarize the provided content.',
  myLargeDocument,
  'Summarize the key points from this document:'
);

if (!decision.fitsInContext) {
  console.log(`Document too large (${decision.inputTokens} tokens). Need to chunk.`);
} else {
  console.log(`OK — ${decision.remainingBudget} tokens remaining for output`);
}
```

### Response Shape

The `countTokens` method returns a `MessageTokensCount` object:

```typescript
interface MessageTokensCount {
  input_tokens: number;  // total tokens the prompt would consume
}
```

Note: The count **may include tokens added by Anthropic internally** for system optimizations
(e.g. internal prompt scaffolding). You are **not billed** for these system-added tokens, but
the count gives you an accurate representation of what will be processed.

### REST API (curl reference)

```bash
curl https://api.anthropic.com/v1/messages/count_tokens \
  --header "x-api-key: $ANTHROPIC_API_KEY" \
  --header "content-type: application/json" \
  --header "anthropic-version: 2023-06-01" \
  --data '{
    "model": "claude-opus-4-5",
    "system": "You are a technical writer.",
    "messages": [
      { "role": "user", "content": "Explain REST APIs." }
    ]
  }'
```

Response:

```json
{ "input_tokens": 23 }
```

## Method 2: The `@anthropic-ai/tokenizer` Package (Legacy — Do Not Use)

### Status: Deprecated for Claude 3+ — Do Not Use in Production

The `@anthropic-ai/tokenizer` npm package (`version 0.0.4`) is an **officially deprecated
local tokenizer**. Anthropic's own README carries a prominent warning:

> "This package can be used to count tokens for Anthropic's older models. As of the Claude 3
> models, this algorithm is no longer accurate, but can be used as a very rough approximation.
> We suggest that you rely on `usage` in the response body wherever possible."

| Property | Status |
|----------|--------|
| Latest version | 0.0.4 |
| Last publish date | Over 1 year ago (as of April 2026) |
| Accurate for Claude 1/2 | Yes |
| Accurate for Claude 3+ | No — rough approximation only |
| Actively maintained | No |
| Beta/stable | Still marked beta |
| Weekly npm downloads | ~49,000 (legacy projects still use it) |

### Why It's Inaccurate

Claude 3+ uses a different BPE tokenizer vocabulary than the one embedded in this package.
The vocabulary mismatch means counts can be off by a meaningful margin on real prompts,
particularly for code, non-English text, and heavily structured content.

### Installation (for reference only)

```bash
npm install --save @anthropic-ai/tokenizer
```

### Usage (for reference only)

```typescript
import { countTokens } from '@anthropic-ai/tokenizer';

// WARNING: Only accurate for Claude 1 and Claude 2 models.
// For Claude 3+, use client.messages.countTokens() instead.
const approximateCount = countTokens('Hello, world!');
console.log(approximateCount); // rough estimate only
```

**Do not use this package for any Claude 3+ project.** Use the API endpoint instead.

## Method 3: Heuristic Estimation (Offline, No API Call)

When you cannot make an API call — for example, during an initial chunking pass before any
network requests are made — heuristic estimation provides a fast offline estimate.

### The Rules of Thumb

| Method | Formula | Typical Accuracy (English prose) |
|--------|---------|----------------------------------|
| Character-based | `chars / 4` | ~85–95% |
| Word-based | `words × 1.3` | ~85–95% |
| Hybrid | `avg(chars/4, words*1.3)` | Slightly better than either alone |

These are based on the statistical properties of BPE tokenization applied to English text.
Accuracy degrades for:
- Code (especially with long identifiers, strings, and symbols)
- Non-Latin scripts (Japanese, Chinese, Arabic — can be 2–4x the character-to-token ratio)
- Heavily formatted content (Markdown, JSON, XML)
- Mixed-language content

### TypeScript Implementation

```typescript
/**
 * Estimates token count using character-based heuristic.
 * Accuracy: ~85–95% for English prose. Use only for quick offline estimation.
 * For accurate counts, use client.messages.countTokens().
 */
function estimateTokensByChars(text: string): number {
  return Math.ceil(text.length / 4);
}

/**
 * Estimates token count using word-based heuristic.
 * Accuracy: ~85–95% for English prose.
 */
function estimateTokensByWords(text: string): number {
  const wordCount = text.trim().split(/\s+/).filter(Boolean).length;
  return Math.ceil(wordCount * 1.3);
}

/**
 * Hybrid estimate — averages char-based and word-based methods.
 * Slightly more stable than either method alone.
 * Still offline/approximate; not suitable for billing-grade accuracy.
 */
function estimateTokensHybrid(text: string): number {
  const charBased = estimateTokensByChars(text);
  const wordBased = estimateTokensByWords(text);
  return Math.ceil((charBased + wordBased) / 2);
}

/**
 * Estimates the full prompt token cost: system + messages.
 * Adds a 10% safety margin on top of the estimate to account for
 * tokenizer variance and Anthropic's internal scaffolding overhead.
 */
function estimateFullPromptTokens(
  systemPrompt: string,
  messages: Array<{ role: string; content: string }>,
  safetyMarginPercent = 10
): number {
  const systemTokens = estimateTokensHybrid(systemPrompt);
  const messageTokens = messages.reduce(
    (sum, msg) => sum + estimateTokensHybrid(msg.content) + 4, // +4 per message for role/metadata
    0
  );
  const raw = systemTokens + messageTokens;
  return Math.ceil(raw * (1 + safetyMarginPercent / 100));
}

// Example usage
const estimate = estimateFullPromptTokens(
  'You are a documentation assistant.',
  [{ role: 'user', content: myLargeDocument }]
);
console.log(`Estimated tokens (with 10% margin): ${estimate}`);
```

### When to Use Heuristics vs. the API

| Scenario | Recommended Approach |
|----------|---------------------|
| Pre-chunking pass on many documents | Heuristic (fast, free, no network) |
| Final check before an important API call | `countTokens()` API |
| Rate-limited environment | Heuristic (countTokens has its own rate limit) |
| Non-English or code-heavy content | Always use `countTokens()` API |
| Development / debugging | Either; heuristic is faster to iterate with |
| Production chunking for accuracy | `countTokens()` API |

**Recommended pattern for the Wiki CLI**: Use heuristics for an initial rough split, then
verify with `countTokens()` before the actual `messages.create()` call if the estimate is
close to the limit (within 20% of the threshold).

## Method 4: The `usage` Field in API Responses

### Overview

Every successful `messages.create()` response includes a `usage` object with the exact counts
for that call. This is the ground truth for billing and is the most reliable source for
post-call tracking and cumulative usage accounting.

### Response Shape (Non-Streaming)

```typescript
import Anthropic from '@anthropic-ai/sdk';

const client = new Anthropic();

const response = await client.messages.create({
  model: 'claude-opus-4-5',
  max_tokens: 1024,
  system: 'You are a technical writer.',
  messages: [{ role: 'user', content: 'What is TypeScript?' }],
});

// Full usage object
console.log(response.usage);
/*
{
  input_tokens: 21,
  output_tokens: 89,
  cache_creation_input_tokens: 0,
  cache_read_input_tokens: 0
}
*/

console.log(`Input:  ${response.usage.input_tokens} tokens`);
console.log(`Output: ${response.usage.output_tokens} tokens`);
console.log(`Total:  ${response.usage.input_tokens + response.usage.output_tokens} tokens`);
```

### Full Usage Object Shape (TypeScript)

```typescript
interface Usage {
  input_tokens: number;                  // tokens from your prompt, not cached
  output_tokens: number;                 // tokens in the model's response
  cache_creation_input_tokens?: number;  // tokens written to prompt cache
  cache_read_input_tokens?: number;      // tokens read from prompt cache (cheap)
}
```

### Cumulative Usage Tracker

For multi-turn conversations or multi-step agentic flows, accumulate usage across calls:

```typescript
interface CumulativeUsage {
  inputTokens: number;
  outputTokens: number;
  cacheCreationTokens: number;
  cacheReadTokens: number;
  callCount: number;
}

class UsageTracker {
  private totals: CumulativeUsage = {
    inputTokens: 0,
    outputTokens: 0,
    cacheCreationTokens: 0,
    cacheReadTokens: 0,
    callCount: 0,
  };

  record(usage: Anthropic.Usage): void {
    this.totals.inputTokens += usage.input_tokens;
    this.totals.outputTokens += usage.output_tokens;
    this.totals.cacheCreationTokens += usage.cache_creation_input_tokens ?? 0;
    this.totals.cacheReadTokens += usage.cache_read_input_tokens ?? 0;
    this.totals.callCount += 1;
  }

  get summary(): CumulativeUsage {
    return { ...this.totals };
  }

  printSummary(): void {
    const t = this.totals;
    console.log(`--- Usage Summary (${t.callCount} calls) ---`);
    console.log(`Input tokens:          ${t.inputTokens.toLocaleString()}`);
    console.log(`Output tokens:         ${t.outputTokens.toLocaleString()}`);
    console.log(`Cache write tokens:    ${t.cacheCreationTokens.toLocaleString()}`);
    console.log(`Cache read tokens:     ${t.cacheReadTokens.toLocaleString()}`);
    console.log(`Total processed:       ${(t.inputTokens + t.outputTokens).toLocaleString()}`);
  }
}

// Usage
const tracker = new UsageTracker();

const r1 = await client.messages.create({ model, max_tokens, messages: turn1 });
tracker.record(r1.usage);

const r2 = await client.messages.create({ model, max_tokens, messages: turn2 });
tracker.record(r2.usage);

tracker.printSummary();
```

### Streaming: Token Usage in Events

When using `stream: true`, usage appears in two places:

1. **`message_start` event** — carries initial `input_tokens` and cache tokens.
2. **`message_delta` event** — carries cumulative `output_tokens`. As of 2025, it also carries
   updated input and cache counts (Anthropic extended the spec; upgrade the SDK if you see
   missing fields).

The SDK's `.stream()` helper accumulates all events automatically. Calling `.finalMessage()`
after streaming completes returns the same complete `Message` object with fully populated
`usage` as a non-streaming call would return.

```typescript
import Anthropic from '@anthropic-ai/sdk';

const client = new Anthropic();

async function streamWithUsage(
  model: string,
  messages: Anthropic.MessageParam[]
): Promise<void> {
  const stream = client.messages.stream({
    model,
    max_tokens: 1024,
    messages,
  });

  // Process text as it arrives
  stream.on('text', (text) => process.stdout.write(text));

  // Get final message with complete usage after streaming finishes
  const finalMessage = await stream.finalMessage();

  console.log('\n\n--- Token Usage ---');
  console.log(`Input tokens:  ${finalMessage.usage.input_tokens}`);
  console.log(`Output tokens: ${finalMessage.usage.output_tokens}`);
}
```

## Estimating Total Prompt Size Before a Call

The following utility combines the API-based approach with a lightweight guard for the
chunking use-case:

```typescript
import Anthropic from '@anthropic-ai/sdk';

const client = new Anthropic();

interface PromptBudget {
  estimatedTokens: number;       // heuristic (offline, fast)
  exactTokens?: number;          // from countTokens API (if called)
  contextLimit: number;          // model's practical input limit
  outputReserve: number;         // tokens reserved for model's response
  effectiveLimit: number;        // contextLimit - outputReserve
  fitsWithEstimate: boolean;     // based on heuristic
  fitsExactly?: boolean;         // based on API count (if called)
  recommendChunking: boolean;
}

const CONTEXT_LIMITS: Record<string, number> = {
  'claude-opus-4-5':             200_000,
  'claude-sonnet-4-5':           200_000,
  'claude-haiku-3-5-20241022':   200_000,
};

const DEFAULT_OUTPUT_RESERVE = 4096;

async function analyzePromptBudget(
  model: string,
  systemPrompt: string,
  userContent: string,
  options: {
    outputReserve?: number;
    useApiCount?: boolean;       // set true for accuracy when near limit
    chunkingThreshold?: number;  // fraction of limit that triggers chunking recommendation (default 0.85)
  } = {}
): Promise<PromptBudget> {
  const outputReserve = options.outputReserve ?? DEFAULT_OUTPUT_RESERVE;
  const threshold = options.chunkingThreshold ?? 0.85;
  const contextLimit = CONTEXT_LIMITS[model] ?? 100_000;
  const effectiveLimit = contextLimit - outputReserve;

  // Fast offline estimate
  const allText = systemPrompt + '\n' + userContent;
  const estimatedTokens = estimateTokensHybrid(allText);
  const fitsWithEstimate = estimatedTokens <= effectiveLimit;
  const nearLimit = estimatedTokens > effectiveLimit * threshold;

  let exactTokens: number | undefined;
  let fitsExactly: boolean | undefined;

  // Only call the API when the estimate is near the threshold — saves rate limit budget
  if (options.useApiCount || nearLimit) {
    const result = await client.messages.countTokens({
      model,
      system: systemPrompt,
      messages: [{ role: 'user', content: userContent }],
    });
    exactTokens = result.input_tokens;
    fitsExactly = exactTokens <= effectiveLimit;
  }

  const recommendChunking = fitsExactly !== undefined
    ? !fitsExactly
    : !fitsWithEstimate;

  return {
    estimatedTokens,
    exactTokens,
    contextLimit,
    outputReserve,
    effectiveLimit,
    fitsWithEstimate,
    fitsExactly,
    recommendChunking,
  };
}

// Helper defined earlier — reproduced here for completeness
function estimateTokensHybrid(text: string): number {
  const charBased = Math.ceil(text.length / 4);
  const wordBased = Math.ceil(text.trim().split(/\s+/).filter(Boolean).length * 1.3);
  return Math.ceil((charBased + wordBased) / 2);
}
```

## Best Practices

1. **Use `client.messages.countTokens()` as the primary method** for any production context
   boundary check. It is free, accurate, and requires no extra headers as of 2025.

2. **Do not use `@anthropic-ai/tokenizer`** for any Claude 3+ project. Its vocabulary does not
   match current models and it is no longer maintained.

3. **Apply heuristics only for initial rough splits** when processing many documents before
   making API calls. Use the `countTokens` API to verify before the actual call, especially
   when close to the context limit.

4. **Add a safety margin** (10–15%) on top of any heuristic estimate to account for variance.

5. **Pass the exact same payload to `countTokens` as you intend to send to `messages.create`**,
   including tool definitions and system prompts. Missing these will undercount.

6. **Track `usage` from every response** for cost accounting. Cache fields
   (`cache_creation_input_tokens`, `cache_read_input_tokens`) are important when prompt caching
   is enabled — they have different billing rates.

7. **For streaming calls**, use `.stream().finalMessage()` to get a single complete `usage`
   object rather than trying to sum `message_start` and `message_delta` events manually.

8. **The `countTokens` rate limit is separate** from the `messages` rate limit. If you are near
   your messages rate limit, you can still call `countTokens` without impacting throughput.

## Common Pitfalls

| Pitfall | Description | Fix |
|---------|-------------|-----|
| Forgetting tool tokens | Tool definitions can add hundreds of tokens. Omitting them from `countTokens` gives an undercount. | Always pass `tools` to `countTokens` if the actual call uses them. |
| Using the legacy tokenizer | `@anthropic-ai/tokenizer` is inaccurate for Claude 3+. | Use `client.messages.countTokens()`. |
| Not reserving output budget | Counting only inputs against the context limit. | Subtract your intended `max_tokens` from the context limit before comparing. |
| Counting once for multi-turn | Each turn adds to the prompt. Counts from turn 1 are stale by turn 3. | Re-run `countTokens` with the full accumulated message history each turn. |
| Ignoring cache tokens in billing | `cache_read_input_tokens` are billed at ~10% the rate of regular input tokens. | Track them separately in your cost model. |
| Heuristics on code/non-English | Ratios are tuned for English prose. Code and CJK text can diverge 30%+ from estimates. | Use the API for any non-prose content. |

| Assumption | Confidence | Impact if Wrong |
|------------|------------|-----------------|
| `countTokens` is GA and requires no beta header as of April 2026 | HIGH | Would need to add `betas: ['token-counting-2024-11-01']` back. Use `client.beta.messages.countTokens()` instead of `client.messages.countTokens()`. |
| Heuristic accuracy of 85–95% for English prose | HIGH | For non-English/code content, accuracy drops significantly; always use the API in those cases. |
| `@anthropic-ai/tokenizer` v0.0.4 is the latest and is deprecated | HIGH | If Anthropic publishes a new version with Claude 3+ support, the picture changes. |
| All listed models have 200K context windows | MEDIUM | Anthropic may adjust limits; verify at the official models page before hardcoding. |
| `usage.cache_creation_input_tokens` and `usage.cache_read_input_tokens` are always present | MEDIUM | They may be absent (`undefined`) when prompt caching is not active; use `?? 0` guards. |

### What Is Explicitly Out of Scope

- **Python SDK** — all examples are TypeScript/Node.js only.
- **Bedrock and Vertex AI deployments** — token counting via those platforms may differ.
- **Token counting for multimodal inputs** (images, PDFs) — image tokens depend on dimensions
  and are not captured by text heuristics. Always use `countTokens` API for image-containing
  prompts.
- **Cost calculation** — pricing rates change; this document does not include dollar amounts.

## Clarifying Questions for Follow-up

1. **Image/PDF content**: Does the Wiki CLI need to process documents that include images or
   embedded PDFs? If so, heuristic token estimation is not applicable for those content types,
   and the `countTokens` API must be used for every chunk.

2. **Rate limit sensitivity**: How many documents will be processed per session? If hundreds of
   large documents are chunked per run, the `countTokens` rate limit could become a bottleneck.
   In that case, a hybrid strategy (heuristics first, API only near threshold) becomes important.

3. **Non-English content**: Will the Wiki CLI process documentation in languages other than
   English? If yes, heuristic accuracy drops significantly and the API-based approach should
   be the default for all content.

4. **Target models**: Are there specific Claude models targeted for this tool? This matters
   for hardcoding context window limits in the chunking logic.

5. **Prompt caching**: Is the tool designed to use prompt caching for repeated base context
   (system prompt, document index, etc.)? If so, the `cache_read_input_tokens` and
   `cache_creation_input_tokens` fields become important for both token budget calculations
   and cost tracking.
